Contexte du projet
------------------

SimplifiedAction est une application web permettant d'analyser une action
financiere de maniere simplifiee pour un public debutant. L'utilisateur peut
rechercher une valeur (nom ou ticker), consulter un score simplifie (0 a 10),
voir des indicateurs (rendement, risque), lire des actualites, gerer ses
favoris et souscrire a une offre premium.

La solution repose sur une architecture client-serveur :
- Frontend : pages statiques HTML/CSS/JavaScript (vanilla)
- Backend : Node.js + Express (API REST)
- Donnees : PostgreSQL (avec mode memoire en fallback)
- Services externes : Finnhub, Yahoo Finance RSS, Stripe


1) ARCHITECTURE

Description
-----------

L'application est organisee en couches :

- Couche presentation : dossier public/ (pages, styles, scripts client)
- Couche API : routes Express (auth, recherche, actualites, favoris, Stripe)
- Couche metier : modules lib/ (calcul score, acces BDD, client Finnhub)
- Couche donnees : PostgreSQL (users, subscriptions, favorites, etc.)

Flux principal :
1. Le navigateur envoie une requete HTTP au serveur Express.
2. Le serveur traite la demande (metier + controle d'acces).
3. Le serveur interroge la BDD et/ou un service externe.
4. Le serveur renvoie une reponse JSON ou une page.

Productions realisees (exemples)
--------------------------------

- Frontend : pages accueil, compte, abonnements, favoris, liste S&P 500.
- API : routes /api/search, /api/news, /auth/*, /api/favorites/*,
  /api/stripe/*.
- Refactor de structure : separation app.js (app exportee) et server.js
  (demarrage), utile pour les tests.

2) SECURITE


Description
-----------

Les mecanismes de securite implementes sont :

- Authentification par JWT : token signe cote serveur.
- Protection des routes sensibles (ex : favoris, profil utilisateur, paiement).
- Hashage des mots de passe avec Argon2 (jamais de mot de passe en clair).
- Secrets stockes dans .env (non versionne via .gitignore).
- Requetes SQL parametrees (placeholders $1, $2...) contre l'injection SQL.
- Verification de signature des webhooks Stripe.

Productions realisees (exemples)
--------------------------------

- Middleware d'authentification reutilisable.
- Route /auth/me avec verification du token.
- Gestion du statut premium via abonnement Stripe en base.

  apres la section "Securite" ou en annexe B.

6) EXPLICATION DETAILLEE DE LA NOTATION DES ACTIONS
===============================================================================

Description
-----------

La notation des actions est concue pour un public debutant, avec une lecture
simple et pedagogique :

- score_simplifie : note de 0 a 10
- rendement : variation en pourcentage (valeur absolue)
- risque : indicateur de volatilite normalise entre 0 et 1

La logique est centralisee dans lib/score.js (fonction computeScore) et
appelee par la route GET /api/search.

Etapes de calcul :
1. Verification des donnees d'entree (quote Finnhub)
   - c : prix actuel
   - dp : variation en pourcentage sur la seance
   - h : plus haut de la seance
   - l : plus bas de la seance
   Si c n'est pas numerique, on retourne une valeur de securite :
   score = 5, rendement = 0, risque = 0.5.

2. Calcul du rendement (indicateur de performance)
   Formule :
   rendement = |dp|
   puis arrondi a 2 decimales.
   Explication simple :
   on garde l'amplitude de la variation (hausse ou baisse), pour montrer
   "combien ca bouge" en pourcentage.

3. Calcul du risque (indicateur de volatilite)
   Formule intermediaire :
   volatilite = (h - l) / c
   Puis conversion en score de risque :
   risque = 0.3 + 2 * volatilite
   Enfin, bornage :
   risque = min(1, max(0, risque))
   Explication simple :
   plus l'ecart entre le plus haut et le plus bas est grand, plus le titre
   est juge instable, donc plus le risque augmente.

4. Calcul du score simplifie (note finale sur 10)
   Formule brute :
   score = 5 + (dp / 10) - (0.5 * risque)
   Puis bornage et arrondi :
   score = min(10, max(0, score))
   score = arrondi(score, 1 decimale)
   Explication simple :
   - base neutre a 5/10
   - bonus si la variation dp est positive
   - malus si le risque est eleve
   Le resultat final reste toujours entre 0 et 10.

Ce choix permet d'obtenir une note lisible rapidement, tout en conservant
une logique metier justifiable (bonus si variation positive, malus si risque
eleve).

Productions realisees (exemples)
--------------------------------

- Module metier dedie : lib/score.js (computeScore, VERSION_ALGO).
- Integration backend : routes/api.js (GET /api/search).
- Gestion des cas de secours :

  sur /api/search).
